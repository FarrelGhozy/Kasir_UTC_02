## 🔴 CRITICAL

### BUG-01: `sendReply()` tidak melempar error — try/catch di dutyReminder.js tidak berguna
- **File:** `backend/bot/dutyReminder.js`, baris 79-83
- **Masalah:** Fungsi `sendReply()` di `wahaClient.js` menggunakan try/catch internal dan **tidak pernah melempar (throw) error** — ia hanya `console.error` dan `return` (baris 11 dan 33-34 di wahaClient.js). Akibatnya, blok `catch (err)` di `dutyReminder.js` baris 81-83 **tidak akan pernah terpanggil**. Jika pengiriman WA gagal, kode akan tetap mencetak `✅ Reminder terkirim` padahal sebenarnya gagal.
- **Dampak:** Tidak ada visibility ketika pengiriman reminder WA gagal. Log monitoring tidak bisa mendeteksi kegagalan.
- **Solusi:** `sendReply()` di `wahaClient.js` harus `throw` error setelah logging, ATAU `dutyReminder.js` harus memeriksa return value. Cara paling mudah: tambahkan `throw error` di blok catch `wahaClient.js` setelah `console.error`.

## 🟠 MAJOR

### BUG-02: `deleteSchedule` — race condition (find lalu delete terpisah)
- **File:** `backend/controllers/dutyScheduleController.js`, baris 192-206
- **Masalah:** Fungsi `deleteSchedule` melakukan `findById(id)` lalu `findByIdAndDelete(id)` sebagai dua operasi terpisah. Ada race condition: jadwal bisa dihapus oleh request lain antara find dan delete, menyebabkan `findByIdAndDelete` mengembalikan `null` tanpa error, dan response tetap "berhasil dihapus" padahal tidak ada yang dihapus.
- **Dampak:** Response menyesatkan. Bisa juga menyebabkan error jika ada operasi lain yang memodifikasi document yang sama.
- **Solusi:** Gunakan `findByIdAndDelete(id)` langsung dan cek return value-nya. Jika `null`, return 404.

```js
const schedule = await DutySchedule.findByIdAndDelete(id);
if (!schedule) {
  return res.status(404).json({ success: false, message: 'Jadwal piket tidak ditemukan' });
}
```

### BUG-03: `updateSchedule` — tidak ada validasi ObjectId untuk parameter `:id`
- **File:** `backend/controllers/dutyScheduleController.js`, baris 131-185
- **Masalah:** Parameter `id` dari `req.params` langsung digunakan dalam `findById(id)` tanpa validasi ObjectId. Jika user mengirim ID yang bukan format ObjectId yang valid (misal: `abc123`), Mongoose akan melempar `CastError`. Meskipun ada global error handler yang menangani `CastError`, response-nya akan 404 dengan message "Sumber daya tidak ditemukan (ID tidak valid)" yang kurang jelas konteksnya.
- **Dampak:** Error message kurang informatif untuk debugging. Konsistensi: `createSchedule` melakukan validasi ObjectId untuk `user`, tapi `updateSchedule` dan `deleteSchedule` tidak.
- **Solusi:** Tambahkan validasi ObjectId di awal fungsi, sama seperti di `createSchedule` (baris 82-85).

### BUG-04: Backup controller tidak mengekspor/mengimpor DutySchedule
- **File:** `backend/controllers/backupController.js`, baris 17-37 (export) dan baris 44-100 (import)
- **Masalah:** `backupController.js` **tidak terganggu** (tidak ada perubahan), artinya `DutySchedule` **tidak ikut di-export maupun di-import** dalam backup/restore. Jika admin melakukan backup lalu restore, semua data jadwal piket akan hilang.
- **Dampak:** Data jadwal piket hilang setelah restore backup. Admin mungkin tidak menyadari sampai reminder tidak terkirim.
- **Solusi:** Tambahkan `duty_schedules` di export data (baris 19-28) dan di import/deleteMany/insertMany (baris 63-83). Import juga perlu menambahkan `DutySchedule.deleteMany({})`.

## 🟡 MINOR

### BUG-05: `duty_role` di User model tidak punya enum constraint
- **File:** `backend/models/User.js`, baris 50-54
- **Masalah:** Field `duty_role` di User model didefinisikan sebagai `String` bebas tanpa enum, sedangkan di `DutySchedule` model, `duty_role` memiliki enum `['Equipment', 'Admin', 'Chief', 'Secretary', 'PDD']`. Ini inkonsistensi — admin bisa set `duty_role` di User dengan nilai apapun (typo, dll).
- **Dampak:** Data `duty_role` di User bisa tidak konsisten dengan `duty_role` di DutySchedule.
- **Catatan:** Ini mungkin intentional karena `duty_role` di User tampaknya hanya field metadata (bukan sumber kebenaran). Tapi perlu didokumentasikan mana yang menjadi sumber kebenaran untuk duty role.

### BUG-06: Sorting `getAllSchedules` menggunakan `day: 1` tapi day adalah string
- **File:** `backend/controllers/dutyScheduleController.js`, baris 24
- **Masalah:** `.sort({ day: 1, created_at: 1 })` akan mengurutkan secara alfabetis: `jumat, kamis, rabu, selasa, senin` — bukan urutan hari dalam minggu.
- **Dampak:** Tampilan jadwal tidak berurutan sesuai hari dalam minggu.
- **Solusi:** Tambahkan field `day_order` (numeric) di model, atau sort di memory menggunakan mapping.

### BUG-07: `getMySchedule` — populate user yang sudah diketahui
- **File:** `backend/controllers/dutyScheduleController.js`, baris 213-230
- **Masalah:** `getMySchedule` melakukan `.populate('user', ...)` padahal `userId` sudah diketahui dari `req.user.id`. Populate di sini tidak menambah informasi yang berguna karena user sudah login dan datanya ada di token.
- **Dampag:** Query tidak perlu (minor performance impact).
- **Solusi:** Hapus `.populate('user', ...)` dari `getMySchedule`, atau biarkan saja jika frontend membutuhkan data user yang lengkap.

### BUG-08: `deleteTechnician` tidak menghapus DutySchedule terkait
- **File:** `backend/controllers/adminController.js`, baris 91-103
- **Masalah:** Ketika teknisi dihapus, `DutySchedule` yang mereferensikan user tersebut tidak ikut dihapus. Ini menyebabkan orphan documents — jadwal piket untuk user yang sudah tidak ada.
- **Dampak:** Reminder WA akan mencoba mengirim ke user yang tidak ada. `populate('user')` akan mengembalikan `null`, dan dutyReminder akan skip dengan log "unknown" (baris 63).
- **Solusi:** Tambahkan `DutySchedule.deleteMany({ user: req.params.id })` di `deleteTechnician`.

### BUG-09: `updateSchedule` — tidak ada validasi ObjectId untuk parameter `:id`
- **File:** `backend/controllers/dutyScheduleController.js`, baris 133
- **Masalah:** Sama seperti BUG-03. Parameter `id` tidak divalidasi sebagai ObjectId sebelum digunakan.
- **Solusi:** Tambahkan validasi `mongoose.Types.ObjectId.isValid(id)`.
